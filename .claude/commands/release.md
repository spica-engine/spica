Release Spica as version `$ARGUMENTS`. Follow every step in order. Do not skip steps or proceed past a user-approval gate without confirmation.

## Step 1 — Pull latest and verify starting state

```bash
git pull
git fetch --tags
```

Report the HEAD commit hash and message. Confirm the working tree is clean (only untracked files like `.claude/` are acceptable — no staged or unstaged changes).

## Step 2 — Determine previous tag and generate release notes

```bash
git tag --sort=-version:refname | head -5
```

Identify the previous tag (the one immediately before `$ARGUMENTS`). Then collect the commits that will be included in this release:

```bash
git log <prev_tag>..HEAD --oneline
```

Format the commits into human-readable release notes grouped by type (feat, fix, chore, etc.). Present them clearly to the user.

## Step 3 — Create and push the tag

```bash
git tag $ARGUMENTS
git push origin $ARGUMENTS
```

Confirm the tag was pushed. This triggers `gh_release.yml` which auto-creates a **draft** GitHub release with generated notes.

## Step 4 — Wait for draft release and ask for approval

Wait ~15 seconds for the `gh_release.yml` workflow to finish creating the draft, then fetch it:

```bash
gh run list --repo spica-engine/spica --limit 5 --json name,status,conclusion,createdAt,headBranch
gh release view $ARGUMENTS --repo spica-engine/spica
```

Present the user with:
- The release notes that were auto-generated by GitHub
- The commits included (from Step 2)

Then **ask the user**: "The draft release `$ARGUMENTS` is ready. Shall I publish it? (This will trigger Docker image builds, npm package publishing, and Helm chart sync.)"

**Do not proceed until the user confirms.**

## Step 5 — Publish the draft

```bash
gh release edit $ARGUMENTS --repo spica-engine/spica --draft=false
```

Confirm the release URL is returned.

## Step 6 — Monitor the release workflow

Find the `Spica Release` workflow run triggered by the publish event:

```bash
gh run list --repo spica-engine/spica --limit 5 --json databaseId,name,status,conclusion,createdAt,headBranch
```

Poll the run ID every 30 seconds until `status` is `completed`:

```bash
gh run view <run_id> --repo spica-engine/spica --json status,conclusion,jobs
```

Report each job's status as it finishes. If `conclusion` is `failure`, immediately run:

```bash
gh run view <run_id> --repo spica-engine/spica --log-failed
```

Analyse the failure, explain the root cause to the user, and suggest a fix. Do not mark the release as complete if any job failed.

## Step 7 — Final verification

Once the workflow completes successfully, verify all three artifacts are actually live:

### Docker Hub
Check all four images exist at the new version:
```bash
curl -s "https://hub.docker.com/v2/repositories/spicaengine/api/tags/$ARGUMENTS" | python3 -c "import json,sys; d=json.load(sys.stdin); print('api:', d.get('name','NOT FOUND'))"
curl -s "https://hub.docker.com/v2/repositories/spicaengine/panel/tags/$ARGUMENTS" | python3 -c "import json,sys; d=json.load(sys.stdin); print('panel:', d.get('name','NOT FOUND'))"
curl -s "https://hub.docker.com/v2/repositories/spicaengine/migrate/tags/$ARGUMENTS" | python3 -c "import json,sys; d=json.load(sys.stdin); print('migrate:', d.get('name','NOT FOUND'))"
curl -s "https://hub.docker.com/v2/repositories/spicaengine/mongoreplicationcontroller/tags/$ARGUMENTS" | python3 -c "import json,sys; d=json.load(sys.stdin); print('mongoreplicationcontroller:', d.get('name','NOT FOUND'))"
```

### npm packages
Check all six packages are published at the new version:
```bash
for pkg in "@spica/cli" "@spica/devkit-auth" "@spica/devkit-bucket" "@spica/devkit-database" "@spica/devkit-identity" "@spica/devkit-storage"; do
  ver=$(npm view "$pkg" version 2>/dev/null || echo "NOT FOUND")
  echo "$pkg: $ver"
done
```

### Helm chart
```bash
curl -s https://spica-charts.storage.googleapis.com/index.yaml | grep -A2 "version: $ARGUMENTS" | head -6
```

## Step 8 — Report

Summarise the release:
- Version released: `$ARGUMENTS`
- Docker images: list each with ✅ or ❌
- npm packages: list each with ✅ or ❌
- Helm chart: ✅ or ❌
- GitHub release URL

If everything is green, declare the release complete. If anything is red, explain what failed and what the next step should be.
